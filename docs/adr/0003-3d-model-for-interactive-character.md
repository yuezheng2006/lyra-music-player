# ADR-0003: 3D模型作为互动角色技术方案

## 状态
已接受

## 背景
互动角色（跳舞小人）需要实现：
- 根据音乐节奏执行流畅的动作
- 表情和动作反映歌曲情绪
- 支持用户点击互动
- 未来支持用户上传自定义形象

技术方案选择：
1. **2D图片 + 骨骼动画**（Spine、DragonBones）
2. **Live2D**（2D立绘 + 物理模拟）
3. **3D模型**（glTF、FBX等）

## 决策
优先采用**3D模型**方案：
- 使用标准3D模型格式（glTF 2.0优先）
- 通过骨骼绑定和动画系统驱动角色动作
- 使用Three.js或类似库渲染
- 初期提供预设3D角色，未来支持用户上传

## 理由
1. **动作流畅性**：3D骨骼动画比2D更自然，尤其是复杂舞蹈动作
2. **表现力强**：支持多角度观看、光影效果，视觉效果更"炫酷"
3. **扩展性好**：
   - 易于添加新动作（导入新动画clips）
   - 支持动态换装、材质替换
   - 未来可添加物理模拟（布料、头发）
4. **用户上传友好**：用户更容易找到或制作3D模型（VRoid、Ready Player Me等工具普及）
5. **不依赖开源特定方案**：标准格式，技术栈自由度高

## 实现细节
- **模型格式**：glTF 2.0（.glb/.gltf），轻量且标准
- **渲染引擎**：Three.js（当前项目已使用）
- **动画系统**：
  - 预设动作库（idle、dance-slow、dance-fast、cheer、sad等）
  - 根据BPM和情绪选择和混合动画
- **交互**：
  - 鼠标点击触发特殊动作（招手、转圈）
  - 未来支持拖拽、手势识别

## 后果
- **正面**：
  - 视觉效果出色，符合"炫酷"定位
  - 动作表现力强，支持复杂编排
  - 扩展性好，支持未来功能
- **负面**：
  - 文件体积比2D大（需要优化加载）
  - 渲染性能要求高（需要测试低端设备）
  - 需要美术资源（初期可用免费模型库）
- **技术债**：
  - 需要建立模型验证机制（用户上传时）
  - 需要性能监控，保证不影响主视觉渲染

## 备选方案
如性能测试发现3D方案在目标设备上表现不佳，降级方案：
1. 简化模型（降低面数、减少骨骼数）
2. 提供"轻量模式"（用户可选择2D替代）
3. 使用预渲染序列帧（特定场景）

## 相关决策
- 与预分析节奏检测（ADR-0001）配合，使用BPM数据驱动动画速度
- 与情绪引擎（ADR-0002）配合，根据情绪选择动画风格
